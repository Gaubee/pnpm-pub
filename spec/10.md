## 第十章：测试策略与质量保证 (Testing & QA)

对于一款直接操作用户高权限凭据和 NPM 核心资产的工具，必须建立严格的三层防线，防止因迭代导致的越权、发包错误或数据损坏。

### 10.1 算法与业务逻辑单元测试 (Unit Tests)

使用 `vitest` 构建极速的底层逻辑防线，无需启动任何真实服务。

1.  **动态密码对齐测试：**
    - Mock 系统时间，给定固定的 Secret，验证 `otplib` 生成的 6 位验证码是否与 RFC 6238 标准的参照表 100% 吻合。
2.  **时钟漂移自愈算法测试：**
    - Mock 拦截到的 `403 OTP validation failed` 错误体。
    - 注入带有时差的 HTTP `Date` Header，验证系统能否精准计算出 `timeOffset`，并生成经过补偿的正确 TOTP。
3.  **Workspace 扫描引擎测试：**
    - 在内存文件系统 (`memfs`) 中构建包含各类 `.gitignore`、`pnpm-workspace.yaml`、嵌套层级和 `"private": true` 的虚拟工程结构。
    - 验证目录扫描器输出的有效包列表是否精准、且无冗余。

### 10.2 系统级沙箱隔离测试 (Integration Tests)

此环节验证 Node.js 进程与底层 OS 设施及配置文件的交互。

1.  **Keychain 凭据隔离读写：**
    - **安全红线：** 绝对不能在测试时污染或覆盖开发者物理机上真实的 `pnpm-pub` 凭据。
    - 编写测试用例时，强制将 `Service Name` 改为 `pnpm-pub-test-sandbox`。测试加密、解密、多 Profile 隔离，并在 `afterAll` 钩子中执行强制擦除。
2.  **导入/导出加密算法测试：**
    - 给定测试的 Profile 数据和密码，验证生成的 JSON 备份文件是否为强加密乱码，且能够使用正确的密码无损还原。

### 10.3 基于 Verdaccio 的全链路模拟测试 (E2E Tests)

使用 E2E 测试验证 CLI、Daemon、IPC 管道、WebUI 和远端 Registry 之间的完整大闭环。

1.  **测试环境搭建：**
    - 在 CI/CD 流水线（如 GitHub Actions）中，通过 Docker 启动一个私有的 `Verdaccio` 实例，模拟真实的 NPM Registry。
2.  **自动化流水线模拟：**
    - **脚本挂载：** 启动 `pnpm-pub` 守护进程。
    - **触发发包：** 脚本执行 `pnpm-pub publish --registry http://localhost:4873`。
    - **验证挂起：** 确保 CLI 进程进入挂起状态，并未向 Verdaccio 泄露任何请求。
    - **模拟 UI 确认：** 编写 Node.js 脚本模拟 SvelteKit 客户端，携带生成的 `WebToken` 向 Daemon 发送 WebSocket 的 `ACTION_CONFIRM_PUBLISH` 指令。
    - **验证结果：** 检查 CLI 是否成功退出（Exit Code 0），并查询 `Verdaccio` 的 API 确认测试包已被成功发布。
