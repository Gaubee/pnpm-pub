## 第二章：系统总体架构 (System Architecture)

### 2.1 整体架构拓扑 (Architecture Topology)

基于轻量化和职责分离的原则，`pnpm-pub` 整体划分为三个核心物理模块：**CLI 客户端**、**Node.js Daemon 守护进程** 和 **Svelte WebUI**。彻底摒弃复杂的本地外挂编译，全部基于 Node.js 生态实现跨平台。

```text
┌───────────────────────┐         IPC (Named Pipe / Unix Socket)        ┌──────────────────────────────────┐
│                       │ ────── [publish cmd + CWD args] ────────────▶ │                                  │
│   Thin CLI Client     │                                               │       Node.js Daemon (Hub)       │
│   (yargs parsing)     │ ◀───── [Stdout/Stderr / Exit Code] ────────── │   (State, NPM API, Auth Core)    │
│                       │                                               │                                  │
└───────────────────────┘                                               │  ┌────────────────────────────┐  │
                                                                        │  │     @github/keytar         │  │
                                                                        │  │  (OS Keychain Native API)  │  │
                                                                        │  └────────────────────────────┘  │
┌───────────────────────┐                                               │  ┌────────────────────────────┐  │
│                       │        HTTP/WS + WebToken Auth                │  │       opentray Native      │  │
│    Svelte WebUI       │ ◀────── [Serve UI Assets + State] ──────────▶ │  │  (Tray Icon & Window Host) │  │
│    (shadcn-svelte)    │ ─────── [User Confirm / TP Event] ──────────▶ │  └────────────────────────────┘  │
│                       │                                               │                                  │
└───────────────────────┘                                               └──────────────────────────────────┘
```

### 2.2 模块职责划分 (Component Responsibilities)

#### 1. Thin CLI Client (瘦客户端)

- **职责：** 纯粹的用户命令行输入代理。
- **行为：**
  - 使用 `yargs` 解析入参（保持与 `pnpm publish` 100% 兼容）。
  - 尝试连接后台 Daemon 的具名管道（Named Pipe / Unix Socket）。
  - 如果连接失败，说明 Daemon 未启动，自动尝试唤起 Daemon（或提示用户运行 `pnpm-pub start`）。
  - 将当前工作目录 (CWD) 和 CLI 参数发给 Daemon，挂起自身，将 Daemon 返回的执行日志实时 `pipe` 到终端终端，直到收到退出码。

#### 2. Node.js Daemon (核心中枢进程)

- **职责：** 全局状态管控、原生系统交互、NPM 业务执行。
- **行为：**
  - **凭据管理：** 启动时通过 `@github/keytar` 向系统凭据管理器（Credential Manager / Keychain）请求解密所有 Profile 的 `Token` 和 `TOTP Secret`，并驻留内存。
  - **验证码生成：** 使用 `otplib` 在内存中实时计算当前 6 位 2FA 验证码。
  - **GUI 宿主：** 引入 `opentray`，接管操作系统的托盘图标逻辑（呈现 NPM Logo + User Avatar），控制原生窗口的 Show/Hide/KeepOnTop。
  - **本地服务：** 开启带有 WebToken 鉴权机制的本地 HTTP 服务，负责为 WebUI 提供静态资源服务及 API 交互。

#### 3. WebUI (交互呈现层)

- **职责：** 负责向用户展示需要物理确认的信息及全局资产看板。
- **行为：**
  - 运行在 `opentray` 唤起的原生窗口内的 WebView 中。
  - 通过 URL Hash / 注入环境变量获取本地 Daemon 的 WebToken，确保所有接口请求安全。
  - 展示侧边栏多账号聚合、发布确认 Diff 对比卡片、Trusted Publishing Event 配置面板等。

### 2.3 技术栈选型清单 (Technology Stack)

为保证最终产物的极致轻量与高可维护性，确立以下技术底座：

- **开发语言：** TypeScript 全栈。
- **构建打包：** `tsdown`。利用其极速内联与压缩特性，将绝大多数 Node.js 依赖直接 Bundle 到最终产物中，减小分发体积与依赖层级。
- **CLI 解析：** `yargs`。成熟稳定，完美胜任 `pnpm publish` 繁多参数的透传与指令解析。
- **系统安全存储：** `@github/keytar`。利用其预编译 (`prebuilds`) 特性直接调用 OS Keychain，替代原先计划的 Rust `napi-rs` 方案，极大降低工程复杂度。
- **TOTP 算法：** `otplib`。用于在内存中基于 Secret 生成动态密码。
- **托盘与窗口控制：** `opentray` (作者自有库)。负责所有跨平台的原生托盘与 KeepOnTop 窗口渲染。
- **前端 UI 框架：** `Svelte` + `shadcn-svelte`。
  - 使用指令 `pnpm dlx shadcn-svelte init --preset b7VW4OwuB6` 初始化。
  - 优势：无虚拟 DOM，打包体积极小，秒级启动；搭配 shadcn 拥有媲美原生应用的极佳桌面端视觉质感。
