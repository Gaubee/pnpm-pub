## 第八章：核心业务时序图 (Core Sequence Diagrams)

本章通过关键工作流的时序逻辑，梳理 CLI、Daemon（Node.js后台）、OS Keychain 和 WebUI 之间的数据交互与闭环。包含首次凭据初始化、资产导入导出、核心发包阻断及容错流。

### 8.1 核心流 A：首次初始化与自动化 Token 申请流 (Onboarding & Auto-Token)

为了免除用户去 NPM 后台手动签发 Token 的繁琐操作，`pnpm-pub` 提供自动化申请流。
**核心原则：** 用户只需提供一次性的 TOTP 密钥及账号密码，系统自动调用 NPM 底层 REST API 换取 Token。随后**立刻从内存中销毁密码**，仅将 Token 和 TOTP 密钥存入底层密钥库。
**风控降级兜底：** 若由于网络或 NPM 官方风控策略导致静默调用 API 失败（如返回 403 或要求验证），WebUI 必须能优雅降级，展示一个备用的输入框，允许用户去浏览器手动生成 Token 并直接粘贴录入。
**实现注意：** 为了实现严格的“密码阅后即焚”，建议前端在传输密码时，以及后端在接收并传递给 NPM API 时，尽量**使用 Buffer** (如 `Buffer.from(password)`)，使用完毕后调用 `buffer.fill(0)` 进行物理覆写擦除，然后再让 GC 回收。这能防止内存 Dump 攻击。

### 8.2 核心流 B：凭据导出与导入流 (Import / Export)

由于 NPM Token 生成后无法再次查看，且重新配置极为繁琐，必须提供带密码保护的跨设备迁移方案。

**导出动作 (Export)：**

1. 用户在 WebUI 请求导出，并输入一个**自定义保护密码 (Protection Password)**。
2. Daemon 从 OS Keychain 提取出所有的 Token 和 TOTP Secret。
3. Daemon 随机生成 `Salt` (盐) 与 `IV` (初始化向量)。
4. 使用**密钥派生函数 (如 PBKDF2 或 scrypt)** 结合 `Salt`，将用户的短密码拉伸为 256 位强密钥。
5. 利用该强密钥，通过 **AES-256-GCM** 对敏感数据进行对称加密。最终生成包含密文、Salt 和 IV 的 `.json` 备份文件。
6. 导出的 JSON 文件结构应包含明文的 Profile 列表元数据，以及加密载荷：`{ "profiles": ["work", "personal"], "salt": "hex...", "iv": "hex...", "ciphertext": "hex..." }`，以便导入时先展示列表再要求解密。

**导入动作 (Import)：**

### 8.4 核心流 D：时钟漂移自愈机制 (Clock Drift Auto-Recovery)

此流为核心流 C（步骤 10 与 11）的容错分支，专门处理 TOTP 强依赖本地时间戳导致的验证失败问题。

1.  **首发失败：** Daemon 携带首次生成的 TOTP 请求 NPM，NPM 返回 `403 OTP validation failed`。
2.  **获取基准：** Daemon 拦截到报错，不立刻抛出。从 HTTP 响应体中提取 NPM 服务器的 `Date` 响应头时间。
3.  **计算偏移：** `timeOffset = NPM_Server_Time - Local_System_Time`。
4.  **应用补偿重试：** 将 `timeOffset` 注入给 `otplib` 的配置中，生成第二颗校准后的动态密码。
5.  **成功或彻底失败：**
    - 带上新密码发起第二次请求。
    - **若成功：** 向 WebUI 和 CLI 返回成功日志，并在 Event 历史中静默追加一条“时钟补偿重试成功”的记录。
    - **若再次失败：** 确定非时钟问题（可能是 Secret 错误或过期），向 CLI 抛出错误，并在 WebUI 记录 Failed Event。
