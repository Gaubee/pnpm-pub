## 第六章：WebUI 与托盘交互设计 (WebUI & Tray Interaction)

本章定义了原生窗口内的视觉结构、状态隔离机制以及两大核心板块 (`Events` 与 `Workspaces`) 的交互逻辑。UI 层全面基于 `SvelteKit` + `shadcn-svelte` 构建。

### 6.1 全局布局与状态隔离 (Layout & State Isolation)

基于 `sidebar-07` 模板构建全局 Layout。工具的核心设定是**“Profile 绝对隔离”**，即不同身份（如个人开源 vs 公司内网）的数据、事件和工作区互不相通。

1.  **左下角 Profile 切换器 (Bottom-Left Switcher)：**
    - 摒弃顶部切换，将多身份选择器固定在侧边栏最底部（展示当前身份的 NPM Avatar + Username）。
    - 点击可呼出 Menu 切换身份。切换后，整个前端应用的 Store/Context 被清空并重新拉取目标 Profile 的数据。
2.  **主导航栏 (Main Navigation)：**
    - **Events (事件流)**：应用的默认首页和绝对核心。
    - **Workspaces (工作区)**：辅助工具，作为其下级，展示被“置顶 (Pinned)”的具体工作区快捷入口。
3.  **主题与原生质感：**
    - 支持亮/暗色模式（跟随系统）。
    - 顶部保留一块极简的 `drag-region` (CSS `app-region: drag`)，允许用户拖拽移动这个原生托盘无边框窗口。

### 6.2 绝对核心：Events 事件中枢 (The Events Hub)

`Events` 页面不仅是历史记录（日志），更是所有交互动作的发起地和审批地。

1.  **事件流视图 (Event Timeline / List)：**
    - 按时间倒序排列卡片流。
    - 事件状态本体包含：**待处理 (Pending)**、**成功 (Success)**、**失败 (Failed)**、**已拒绝 (Rejected)**、**凭据过期 (Expired)**、**需要用户动作 (Action Required)**。其中 `Expired` / `Action Required` 是凭据续期作用源，不应退化为普通失败投影。
2.  **待处理审批机制 (Pending Action & Interception)：**
    - 当 CLI 触发发包被后台 Daemon 拦截时，生成一个 `Pending Event`。
    - 此时托盘闪烁，窗口 `KeepOnTop` 弹出并自动路由至 `Events` 页面，高亮显示该待处理卡片。
    - **上下文穿透 (Context Override)：** 如果用户在终端显式传入了 `--profile` 参数，引发拦截的 Event 可能会与当前 UI 左下角选中的 Profile 不一致。此时该待处理卡片必须**打破隔离**，不仅展示 Diff（包名、新旧版本号），还必须**高亮显示执行此动作的强制身份标识（Avatar + Username）**，以防用户混淆。
    - 卡片底部提供显眼的 **[Confirm Publish]** 和 **[Reject]** 按钮。
    - 如果用户点击了 [Reject]（或者直接关闭了托盘窗口），后台 Daemon 必须通过 IPC 立即向挂起等待的 CLI 客户端发送一个 SIGINT 或 Exit Code 1（并附带消息 "Publish canceled by user."），使终端优雅结束，而不是让终端一直死等。
3.  **主动操作入口 (Proactive Actions)：**
    - 页面顶部提供类似 `[+ New Action]` 的快捷入口，允许用户脱离命令行，直接在 GUI 发起“创建新包 (v0.0.0 占坑)”、“配置 Trusted Publishing”、“批量配置 Trusted Publishing”、“强制刷新本地 Token”等动作，执行后即成为一条新的 Event 沉淀在列表中。
    - Trusted Publishing 的 Dialog 只承担投影和意图收集：已有 Current 配置时弹出当前配置对话框（`Update` / `Remove`）并提供 OIDC workflow 预览/复制/本地写入 tab；批量操作时弹出可反选的包列表与当前配置预览。真正的 npm `/trust` 写操作必须落到 `configure-trust` Pending Event，批量则创建 N 个共享 `groupId` 的 `configure-trust` Event，等待用户在 Events 中逐个确认。
4.  **全局异常信息**
    - 如果后台捕获到 `401 Unauthorized`，这不应该只是一个普通的“Failed Event”，而应该是一个特殊的 “Expired Event (凭据过期)”。UI 此时可以自动弹出一个输入框，提示：“Token 已过期，请输入 NPM 密码以一键续期（密码依然阅后即焚）”，复用我们在 8.1 章设计的“自动化 Token 申请流”。

#### 6.2.5 Trusted Publishing 只读展示 (Readonly Variants)

凡是需要**只读展示**一个已配置的 Trusted Publisher（GitHub Actions / GitLab CI / CircleCI）的地方，统一使用 `TrustedPublishingReadonly` 组件（`webui/src/lib/components/trusted-publishing-readonly.svelte`）。该组件提供三个变体，由 `mode` prop 切换；字段元数据与编辑表单 `TrustedPublishingDraftForm` 共享自 `webui/src/lib/trusted-publishing.ts`，确保两套视图永不偏移。

1.  **变体定义：**
    - **紧凑 `compact`（单行）**：在一行内渲染 provider 名 + 各字段值 chip；每个 chip 用 Tooltip 描述其字段（label + help）。支持基于容器查询的响应式：极端窄宽下 GitHub / GitLab 可省略 Organization/Namespace，仅保留仓库主体。
      - **特例（CircleCI）**：紧凑模式下**只显示 `Version Control System Origin`**，因为其余字段（org-id / project-id / pipeline-definition-id）是不可读的 UUID，单行内展示毫无意义。
    - **多行 `multiline`**：每个字段独占一行，形如 `Label: Value`。
    - **详细 `detailed`**：在多行的基础上，每个字段追加 description，视觉上与编辑表单模式高度一致（同款 `bg-muted/20` 卡片、同款 `text-[10px]` help 文本）。

2.  **各落点的变体选择（决策原话，作为单一事实来源）：**

    > 在 WorkspaceDetail 面板中点击 OIDC 弹出的 Dialog 中，使用多行模式。
    > 这里的决策逻辑是：已经有配置，那么查阅者更多的目的是了解已经配置了哪些字段。而因为渲染在 Dialog 中，空间充足，所以使用多行模式。
    >
    > 在 EventDialog 中，使用详细模式。
    > 这里的决策逻辑是：这里可能会做编辑，因此了解这些字段的用途，然后展开编辑，是很有必要的，所以把字段描述也渲染出来。
    >
    > 其它使用紧凑模式。
    > 这里的决策逻辑是：在非 Dialog 中，那么基本就是在列表中，列表的重点在于每一项 item 不要太高，所以使用单行渲染。

3.  **列表场景的补充约定：** Packages 列表 / PackageDetail / Workspaces 卡片这些“纯列表”场景，仍保留 `TrustedPublishingStatus`（shield 图标 + 状态色 + Configure 按钮的状态指示器外壳），其文本统一由 `trustedPublisherSummary()` 产出。`TrustedPublishingStatus` 与 `TrustedPublishingReadonly` 互补：前者承担“状态色 / loading 动画 / Configure 动作”的外壳语义，后者承担“字段结构化展示”的内容语义。紧凑变体的 chip 化展示目前只落地在 Dialog 内的批处理行；若未来列表项也需要 chip 化，可平滑切换。

#### 6.2.6 Trusted Publishing 分组继承模型 (Group Inheritance)

批量 `configure-trust` 事件共享一个 `groupId`。组内的“默认配置”只存**一份**，由分组默认表单编辑；成员通过**显式的继承标记**决定是用组默认（inherit）还是自带配置（custom）。这套模型解决了一个关键的性能与正确性问题。

> **核心动机（决策原话）：** 继承本身意味着，我们不会对成员的表单做任何修改，只有在最终确认要发布的时候，才会真的落地。而界面相关的显示，也始终遵循“继承”的概念。

1.  **数据模型（daemon 侧，单一事实源）：**
    - `groupTrustDefaults: Map<groupId, TrustedPublisherCreateConfig>` —— 组默认 config，**全组只有一份**。
    - `groupInheritMembers: Map<groupId, Set<eventId>>` —— 显式标记组内哪些成员**继承**组默认。`config` 字段是否为空**不**用于推断继承（那是把两个职责耦合在一起）；继承是一个独立的、显式的标记。
    - 成员的 `payload.data.config` 只在 **custom** 成员上被写；inherit 成员的 config 永远不被默认表单触碰。

2.  **为什么不能 fan-out（历史教训）：** 早期实现里 `updateConfigureTrustGroupDraft` 会把默认 config **复制写进组内每一个成员**，并给前端逐成员回声。这在分组默认表单每次按键时触发 N 次回声 + N 次全卡片重绘，造成 60~100% CPU 占用、输入闪回清空。继承模型下，默认表单只更新组默认这一份、只广播一帧轻量的 `group-trust-draft`，**零成员回声、零 N 倍放大**。

3.  **解析规则（单一职责）：** `resolveConfigureTrustConfig(event)`：
    - inherit 成员 → 返回 `groupTrustDefaults[groupId]`；
    - custom 成员 → 返回成员自身的 `payload.data.config`。
      显示侧（成员行 summary、EventDetailDialog 只读视图）与确认侧（`runConfigureTrust`）统一走这个解析，保证“界面显示即确认时落地的值”。

4.  **协议：** 新增 WS 帧 `group-trust-draft`（携带 `groupId` + `defaultConfig` + `inheritMembers`），**单帧**，不携带成员事件。新增 RPC `events.setMemberInherit({ eventId, inherit })`，EventDetailDialog 的 Inherit/Custom 切换调用它。hello/重连快照附带所有 pending 组的 defaults + inheritMembers，刷新不丢草稿与标记。

5.  **生命周期：** 组内最后一个 pending 成员 resolve/reject/fail 时，daemon 清理该 groupId 的两份状态，保持内存有界。

### 6.3 辅助视窗：Workspaces 工作区看板 (The Workspace View)

`Workspaces` 页面的核心使命，是从“本地磁盘路径”的视角，辅助用户更便捷地发起 Event。

1.  **目录引擎与解析提速 (Workspace Parsing)：**
    - 后台 Daemon 自动记录用户执行过发包命令的基准根目录。
    - **性能优化策略：** 扫描目录时，**最高优先级寻找 `pnpm-workspace.yaml`**。如果存在，直接解析其 `packages` 字段定义的 glob 规则去精准定位包目录；如果不存在，再降级为启发式硬遍历（排除 `node_modules`、`.git` 等）。
2.  **资产卡片展示 (Package Cards)：**
    - 经过 Profile 归属过滤后，列出该根路径下的所有有效公开包。
    - 每个包渲染为一个卡片，展示其 `name`、`version`、`description`。
3.  **作为 Event 的触发器 (Action Triggers)：**
    - 在卡片上悬浮时，提供快捷操作：【发布新版本】、【配置 Trusted Publishing】。
    - **核心逻辑闭环：** 点击这些按钮，**并非在 Workspace 页面就地执行**，而是携带该包的上下文，在系统内部生成一个新的 Event，UI 随即自动跳转回 `Events` 页面进入事件处理流。

### 6.4 托盘宿主行为 (Tray Host Behaviors)

`opentray` 承担了系统底层的生命周期与窗口表现：

- **平时状态：** 常驻内存，UI 窗口 `Hide`（不占用任务栏和屏幕空间）。
- **按需呼出：** 用户鼠标点击托盘图标 -> 窗口 `show`，但首次启动与后续重新呼出都必须先把原生透明度锁定为 `0.1`，再由 WebUI 的单一 `WebAnimation` 真源在 1s 内从 `0.1` 映射到 `navigator.opentrayWindow.setStyle({ opacity: 1 })` 完成进入动画。
- **失焦退场：** 当 `canAutoClose = !pinKeepOpen && !hasActiveEvents` 成立且窗口发生 Blur，Daemon 只投影 `exitRequested`；WebUI 以同一个不可见 DOM `animate()` 驱动 6s 退场曲线和 5→0 倒计时。退场曲线由三个 0.5Hz 脉冲组成：`1→0.8→0.9`、`0.9→0.5→0.7`、`0.7→0`；前 4s 脉冲使用类正弦 ease-in-out 呼吸曲线，最后 `0.7→0` 使用退场 settle 曲线；`animation.finished` 后再回调 Daemon 执行真正的 `hide()`。
- **强制唤起 (KeepOnTop / ActiveEvents)：** `pinKeepOpen` 与 `hasActiveEvents` 是正交门控。任一为真都会打破 Blur 退场规则；一旦产生带有交互需求的 `Pending Event`，窗口会被唤起并保持置顶，直至用户处理完该 Event 才恢复正常挂起逻辑。
- **退出进程 (Quit / daemon stop)：** `hide()` 只是窗口投影变化；托盘 `Quit`、`pnpm-pub daemon stop`、版本自杀、SIGINT/SIGTERM 是进程生命周期作用源。Daemon 销毁 tray/window surface 后必须退出进程，让 OpenTray broker 的 session socket 关闭并释放单会话租约，避免下一次 `pnpm dev` 退化为 headless。
