## 第一章：产品愿景与需求定义 (Product Vision & Requirements)

### 1.1 产品定位与核心痛点 (Core Pain Points)

本项目的诞生源自开发者（作者）极其强烈的个人真实痛点，原始需求定义如下：

> **原始意图：**
> “我有一个痛点，就是我有非常多的 npm 包需要发布，但是经常需要操作两步验证 (2FA)。所以我想开发一个工具（叫 `pnpm-pub`），来帮助我实现无感的、快速的发布操作及相关的管理需求。”
> “我的发包需求其实非常简单，通常就是发一个 v0.0.0 的版本，然后自动配置 Trusted Publish。这是我非常常见的需求。”

**产品定位：** `pnpm-pub` 是一个面向高频发包者（NPM Package Authors）的**桌面级本地守护代理（Desktop Agent）与提效工具**。它通过结合本地安全存储与原生托盘 GUI，将极其繁琐的“获取并输入 6 位 2FA 动态码”流程，降维打击为“在置顶窗口点一下鼠标确认”，同时提供 NPM 资产和身份的中枢管理能力。

### 1.2 核心设计哲学 (Core Design Philosophy)

为确保开发过程不发生路线偏移，确立以下不可妥协的设计哲学：

1.  **极速且无感（DX First）：**
    > “只会在启动的时候去获取一次（系统密钥），这时候是需要进行硬件级别的解密。后续这个密钥就一直存放在内存中的...往后就直接通过 GUI 进行点击确认。”
    - **开发约束：** 绝不能让用户在每次发包时都去输密码或按指纹。底层进程必须保持内存驻留（Daemonize），确保“用完即走，随叫随发”。
2.  **安全阻断（Security via Physical Interaction）：**
    > “需要我本地手动执行 pnpm-pub 去发布... 是需要到 GUI 上点击确认才能发布的。以免 AI 误操作执行了这种动作进行恶意发布。”
    - **开发约束：** 哪怕是发 `v0.0.0` 的占坑包，也**绝不允许静默发布**。每一次向 NPM Registry 的 Write 操作，都必须经过 GUI 原生窗口的物理点击授权。
3.  **克制与中立（Unopinionated & Safe Default）：**
    > “占坑的限制就不用刻意提醒了，这种属于个人行为。”
    > “生成（OIDC）结果仅供参考。不同项目有自己的复杂配置。所以文件如果已经存在，就默认不要覆盖了。除非提供 `--force`。”
    - **开发约束：** 工具绝不能自作聪明地干预 NPM 官方的报错，也不能野蛮覆盖用户的本地工程文件。工具的角色是“顺滑的执行者”，而非“傲慢的独裁者”。
4.  **架构职责分明（Keep it Simple & Small）：**
    > “Rust 我只打算用它来封装系统接口，最终还是要在 Node 这里完成最终接口的组装... 否则 Rust 会变得非常复杂。”
    - **开发约束：** 系统原生调用（如 `@github/keytar`）仅作为底层基建；所有业务流、HTTP/IPC、UI 调度全归 Node.js 控制。

### 1.3 核心功能清单 (Features)

1.  **兼容原生 CLI 的瘦客户端：**
    - 提供 `pnpm-pub` 命令，其核心发布动作 (`pnpm-pub publish`) 与原生的 `pnpm publish` 参数**完全兼容**。客户端本身不发包，仅作为向本地 Daemon 发送指令的触发器。
2.  **基于 Opentray 的原生中枢 GUI：**
    - > “使用 tray 技术（我有一个 opentray 的包），实现将 tray 作为 GUI 的入口，点击 tray 就可以出现一个原生的窗口... keepOnTop，自动 hide 和 show。”
    - 常驻系统托盘，图标采用 `NPM Logo + User Avatar` 的合并方式展示。
    - 提供 Sidebar（侧边栏）聚合所有 Profile 身份，支持丝滑的账号切换。
3.  **自动/辅助配置 Trusted Publish (OIDC)：**
    - 提供 CLI/GUI 一键生成 GitHub Actions 脚本（`publish.yml`）的能力，并自动向 NPM 申请绑定仓库。
    - **带防呆机制：** 如果本地已存在该文件，抛出警告，阻断执行，或引导用户使用 `--force`，绝不盲目覆盖。
4.  **Multi-Profile（多身份管理）：**
    - 支持管理多个 NPM 身份（如个人开源账号、公司工作账号）。
    - GUI 上支持快速切换；CLI 发包被拦截弹出 GUI 时，允许在最后一刻通过选单重新确认/切换发包身份。
5.  **自动化凭据与跨设备迁移（跨平台安全基建）：**
    - **零心智初始化：** 用户仅需提供账号、密码与 TOTP 密钥，系统自动代理申请底层长效 Token 并立即销毁内存中的密码。凭据通过底层原生接口安全存储至 Windows Credential Manager 或 macOS Keychain。
    - **安全导入/导出：** 提供带自定义密码保护的 `.json` 资产导出功能，解决 Token 生成后无法二次查看导致的跨设备迁移痛点。

### 1.4 不做的事 (Out of Scope / Non-Goals)

为了保证项目可控与精简，明确以下事项**绝对不在本期或未来的规划内**：

1.  **不支持 Linux 桌面端：** 初期仅将 `Windows` 和 `macOS (Darwin)` 的二进制文件打包在一起。无需为 Linux 的各种复杂桌面环境（GNOME/KDE/密钥环）耗费精力。
2.  **绝不将密码落盘持久化：** 尽管系统提供“首次自动化 Token 申请流”，但用户的 NPM 账号密码仅用于瞬间的接口交换，**绝不允许在本地以明文或任何加密形式持久化存储**。后续所有的验证均依赖硬件存储的 Token 与 TOTP Secret。
3.  **不在客户端模拟 NPM 服务端规则校验：** 不在本地判断包名是否相似、是否会被 NPM 拒绝。一切以向 NPM 发起请求后返回的 Error / 403 / 404 为准，工具只负责透传结果。
4.  **不在 C++/底层引入任何业务逻辑：** 诸如 `@github/keytar` 的 Native Addon 仅当作“纯粹的 K-V 读写接口”使用，所有业务逻辑必须上浮至 Node.js 侧处理。
