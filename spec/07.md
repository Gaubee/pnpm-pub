## 第七章：CLI 客户端设计 (Thin Client Design)

本章定义了 `pnpm-pub` 命令行入口的行为规范。CLI 客户端被严格限制为一个**无状态的瘦客户端 (Thin Client)**，其核心任务只有两个：指令路由与 IPC 通信代理。

### 7.1 CLI 解析策略与 Fallback 路由 (Parsing & Routing)

CLI 入口基于 `yargs` 构建，采用“显式子命令为辅，隐式 Publish 为主”的路由解析机制。

1.  **显式管理命令 (Management Sub-commands)：**
    - `pnpm-pub start [--profile=*]`：主动拉起 Daemon 进程并唤出托盘。
    - `pnpm-pub status`：检查当前后台 Daemon 的运行状态及激活的 Profile。
    - `pnpm-pub stop`：安全关闭并退出 Daemon 进程。
    - `pnpm-pub version`：打印 pnpm-pub **自身** 的版本号（来自 `package.json`），退出码 0，不经过任何 IPC。
2.  **隐式默认路由 (The Fallback Mechanism)：**
    - 当 `yargs` 解析到未知的 command，或者没有 command 时，**自动降级将其视作 `publish` 命令的参数**。
    - **等价示例：**
      - 输入 `pnpm-pub --access public` === 等价于执行 `pnpm publish --access public`。
      - 输入 `pnpm-pub --dry-run --no-git-checks` === 等价于执行 `pnpm publish --dry-run --no-git-checks`。
      - 输入 `pnpm-pub -r --no-git-checks` === 等价于执行 `pnpm publish -r --no-git-checks`（短别名同样兼容）。
    - 这保证了与原版 `pnpm` CLI 100% 的肌肉记忆兼容。

#### 7.1.1 终端意图 (Terminal Intents) —— 原生透传

少数意图永远不会经过 Daemon，而是由 CLI 直接原样转发给 `pnpm`，以保证输出与原版完全一致：

- `pnpm-pub --help` / `pnpm-pub -h` → `pnpm publish --help`。
- `pnpm-pub --version` → `pnpm publish --version`（≡ `pnpm --version`，打印 **pnpm** 的版本）。

> 注意 `--version` 与 `version` 子命令（§7.1）的区分：前者是 publish 终端意图，打印 pnpm 版本；后者是 pnpm-pub 元命令，打印 pnpm-pub 版本。

#### 7.1.2 未知 flag 的「默认转发」策略 (Forward-by-default)

pnpm-pub 的目标是**完整替代** `pnpm publish`，因此对 argv 采取**默认转发**而非默认拒绝：

- Daemon **只拦截**那些它自身用于路由 / 取值的 flag：
  - 路由类：`--recursive`/`-r`/`--multi`/`-m`（递归发布事件）、`--dry-run`（dry-run 路径）、`--filter`/`-F`、`--filter-prod`（递归选择器）。
  - 取值类：`--access`、`--tag`、`--otp`、`--registry`、`--publish-branch`、`--dir`/`-C`、`--reporter`、`--loglevel`、`--changed-files-ignore-pattern`、`--test-pattern`（用于正确识别位置参数 / 包源）。
- **其余所有 flag 原样转发**给 `pnpm publish` 子进程，由 pnpm 本身负责校验与执行。这包括但不限于：`--force`/`-f`、`--batch`、`--skip-manifest-obfuscation`、`--provenance`、`--ignore-scripts`、`--json`、`--report-summary`、`--workspace-root`/`-w`、`--reporter`/`-s`、`--yes`/`-y`，以及任何 pnpm 未来新增的 flag。
- 这种策略对 pnpm 的版本演进天然鲁棒：pnpm 新增 flag 时 pnpm-pub 无需改动即可透传；真正无法识别的 flag 由 pnpm 报错，行为与原版一致。

#### 7.1.3 全局选项 `-C`/`--dir`（cwd 覆盖）

`-C <path>` / `--dir <path>` 是 pnpm 的全局选项，会改变 publish 的工作目录。pnpm-pub 在 `intercept` 阶段解析该选项，将有效 cwd 应用到工作区收集、包源解析与递归枚举；并从转发 argv 中**剥离**该选项（与 `--otp`/`--registry` 的剥离方式一致），因为子进程的 `cwd` 已被权威设置，避免 pnpm 二次解析。

#### 7.1.4 `--provenance` 的转发 + 提示

`--provenance` 触发 SLSA provenance 签名，正常只在受支持的 CI（如 GitHub Actions）+ OIDC 环境下生效。pnpm-pub **原样转发**该 flag（保持 drop-in 行为），同时在 stderr 输出一次性提示，引导需要从 CI 进行可信发布的用户使用 pnpm-pub 的 OIDC Trusted-Publish 流程（见第八章）。

### 7.2 核心业务流：透传与挂起 (Passthrough & Suspend)

当 CLI 命中 Publish 意图时，将执行以下标准化流程：

1.  **IPC 连接建立：**
    - CLI 尝试连接本地约定的 Socket (Unix Domain Socket: `~/.pnpm-pub/run/pnpm-pub.sock` 或 Windows Named Pipe: `\\.\pipe\pnpm-pub-sock`)。
    - **静默拉起 (Auto-Booting)：** 若连接被拒（说明 Daemon 未运行），CLI 会静默执行 `spawn` 拉起幽灵进程（脱离 TTY），轮询等待 Socket 就绪后再重新连接。
    - CLI 向 Daemon 发送的握手指令中带上自身的 version (读取 package.json)。如果 Daemon 发现 CLI 版本高于自身，应该主动返回一个特殊的断开信号，并执行 `process.exit(0)` 自杀。CLI 发现连接断开后，自然会触发 Spawn 拉起 机制，从而唤醒全新版本的 Daemon。
2.  **指令封包发送：**
    - 向 Daemon 发送包含完整上下文的 JSON 载荷：
      ```json
      {
        "command": "publish",
        "cwd": "/absolute/path/to/project",
        "args": ["--access", "public"],
        "profileOverride": "work" // 如果命令行显式传入了 --profile
      }
      ```
3.  **终端静态挂起与 UX：**
    - 发送成功后，CLI 在终端打印一行静态的提示语，保持极简且不破坏终端缓冲：
      `> Waiting for GUI confirmation. Please check your system tray...`
    - 随后 CLI 进程进入 `await` 挂起状态。
4.  **日志代理与退出码透传 (Stdio Proxy)：**
    - Daemon 在 GUI 获得确认并真实调用 NPM API/pnpm 核心包后，会通过 IPC 通道将产出的 `stdout` 和 `stderr` 实时传回。
    - CLI 客户端将这些数据流一字不差地 `pipe` 到当前的终端面板中。
    - 最终，Daemon 返回一个退出码 (Exit Code)，CLI 使用该退出码执行 `process.exit(code)`，完美融入任何外部 Shell 脚本的链路中。
