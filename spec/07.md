## 第七章：CLI 客户端设计 (Thin Client Design)

本章定义了 `pnpm-pub` 命令行入口的行为规范。CLI 客户端被严格限制为一个**无状态的瘦客户端 (Thin Client)**，其核心任务只有两个：指令路由与 IPC 通信代理。

### 7.1 CLI 解析策略与 Fallback 路由 (Parsing & Routing)

CLI 入口基于 `yargs` 构建，采用“显式子命令为辅，隐式 Publish 为主”的路由解析机制。

1.  **显式管理命令 (Management Sub-commands)：**
    - `pnpm-pub start [--profile=*]`：主动拉起 Daemon 进程并唤出托盘。
    - `pnpm-pub status`：检查当前后台 Daemon 的运行状态及激活的 Profile。
    - `pnpm-pub stop`：安全关闭并退出 Daemon 进程。
2.  **隐式默认路由 (The Fallback Mechanism)：**
    - 当 `yargs` 解析到未知的 command，或者没有 command 时，**自动降级将其视作 `publish` 命令的参数**。
    - **等价示例：**
      - 输入 `pnpm-pub --access public` === 等价于执行 `pnpm publish --access public`。
      - 输入 `pnpm-pub --dry-run --no-git-checks` === 等价于执行 `pnpm publish --dry-run --no-git-checks`。
    - 这保证了与原版 `pnpm` CLI 100% 的肌肉记忆兼容。

### 7.2 核心业务流：透传与挂起 (Passthrough & Suspend)

当 CLI 命中 Publish 意图时，将执行以下标准化流程：

1.  **IPC 连接建立：**
    - CLI 尝试连接本地约定的 Socket (Unix Domain Socket: `~/.pnpm-pub/run.sock` 或 Windows Named Pipe: `\\.\pipe\pnpm-pub-sock`)。
    - **静默拉起 (Auto-Booting)：** 若连接被拒（说明 Daemon 未运行），CLI 会静默执行 `spawn` 拉起幽灵进程（脱离 TTY），轮询等待 Socket 就绪后再重新连接。
    - CLI 向 Daemon 发送的握手指令中带上自身的 version (读取 package.json)。如果 Daemon 发现 CLI 版本高于自身，应该主动返回一个特殊的断开信号，并执行 `process.exit(0)` 自杀。CLI 发现连接断开后，自然会触发 Spawn 拉起 机制，从而唤醒全新版本的 Daemon。
2.  **指令封包发送：**
    - 向 Daemon 发送包含完整上下文的 JSON 载荷：
      ```json
      {
        "command": "publish",
        "cwd": "/absolute/path/to/project",
        "args": ["--access", "public"],
        "profileOverride": "work" // 如果命令行显式传入了 --profile
      }
      ```
3.  **终端静态挂起与 UX：**
    - 发送成功后，CLI 在终端打印一行静态的提示语，保持极简且不破坏终端缓冲：
      `> Waiting for GUI confirmation. Please check your system tray...`
    - 随后 CLI 进程进入 `await` 挂起状态。
4.  **日志代理与退出码透传 (Stdio Proxy)：**
    - Daemon 在 GUI 获得确认并真实调用 NPM API/pnpm 核心包后，会通过 IPC 通道将产出的 `stdout` 和 `stderr` 实时传回。
    - CLI 客户端将这些数据流一字不差地 `pipe` 到当前的终端面板中。
    - 最终，Daemon 返回一个退出码 (Exit Code)，CLI 使用该退出码执行 `process.exit(code)`，完美融入任何外部 Shell 脚本的链路中。
