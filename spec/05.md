## 第五章：Node.js Daemon 守护进程设计 (Daemon Core Design)

本章定义了系统的中枢神经——Node.js 守护进程的生命周期、通信基建以及本地工程资产的扫描策略。

### 5.1 进程的保活与按需唤醒 (Lifecycle & Auto-Activation)

为了贯彻“极致无感”的设计哲学，系统不允许强制要求用户手动执行 `start` 命令。守护进程必须具备“按需幽灵启动”的能力。

1.  **按需唤醒机制 (On-Demand Activation)：**
    - 任何 CLI 指令（如 `pnpm-pub publish` 或单纯的 `pnpm-pub` 试图打开 WebUI）被触发时，CLI 客户端首先尝试连接本地的 IPC Socket。
    - 如果连接失败（抛出 `ENOENT` 或 `ECONNREFUSED` 错误），CLI 客户端会截获该错误，并主动通过 `child_process.spawn` 启动真正的 Daemon 进程。
2.  **幽灵进程化 (Detached Process)：**
    - 启动 Daemon 时，CLI 必须传入 `detached: true` 和 `stdio: 'ignore'`（或重定向到 `~/.pnpm-pub/logs/daemon.log`），随后调用 `child.unref()`。
    - 这样可以完全脱离当前终端的 TTY 会话，使用户关闭终端窗口后，Daemon 依然能在后台继续运行并响应托盘请求。
3.  **单例锁 (Single Instance Lock)：**
    - Daemon 启动的第一步是尝试绑定固定的 Named Pipe (Windows) 或 Unix Domain Socket (macOS / Linux)。
    - 如果端口被占用，说明已有幽灵进程存活，当前 Daemon 立即自动退出，确保系统中永远只有一个中枢在运作。

### 5.2 核心服务基建 (Core Services Infrastructure)

Daemon 进程内存驻留后，将同时拉起以下三个核心通信服务：

1.  **IPC Socket 服务器：**
    - 监听本地具名管道，负责接收 CLI 客户端传来的 `command`、`cwd`、`args` 数据载荷。
2.  **WebUI HTTP 服务器：**
    - 强烈建议使用动态随机端口 (port: 0)
    - 基于 `fs` 模块返回 `dist/webui` 目录下由 SvelteKit 编译好的 SPA 静态资源。
3.  **oRPC WebSocket 双向通信服务：**
    - 与 HTTP 挂载在同一端口。
    - 挂载在 `/ws/rpc`，启动时在内存中生成一个 256 位强随机、64 个十六进制字符的 `WebToken`。通过校验升级请求中的 Token 来阻断任何非授权的本地访问；`/api/*` 不再承载 WebUI 业务 API。

### 5.3 本地工作区引擎 (Local Workspace Engine)

根据“自动收集与主动管理”的原则，Daemon 将承担本地代码资产的扫描与过滤工作。为了防止 Monorepo 的子包产生大量零碎的工作区记录，必须具备寻根能力和风险目录防御机制。

1.  **自动收集与寻根机制 (Auto-Collect & Find Root)：**
    - 当收到 CLI 传来的 CWD 时，Daemon **不会直接盲目记录该零碎路径**，而是向上遍历查找基准根目录（优先级：最近的 `pnpm-workspace.yaml` -> `.git` 目录 -> 包含顶层 `package.json` 的目录）。
2.  **风险目录拦截与警告 (Risk Boundary Defense)：**
    - 如果向上遍历直至系统根目录（`/` 或 `C:\`），依然没有发现上述任何一种工程标识文件（例如用户误在 `~/Downloads` 目录下执行了命令）。
    - 此时 Daemon 会挂起该次自动收集动作，并通过 CLI 抛出警告 / GUI 弹出二次确认提示：“未检测到 Git 或 NPM 工程标识。将此庞大目录加入 Workspace 可能会导致操作系统卡顿。您非常确定要添加吗？”
    - **只有在用户明确确认后**，该路径才会被写入配置。这彻底阻断了全局盘符扫描导致的性能灾难。
3.  **工作区存储结构 (`workspaces.json`)：**
    ```typescript
    interface WorkspacesConfig {
      paths: Array<{
        path: string; // 绝对根路径 (Root Path)
        pinned: boolean; // 是否被用户置顶（收藏）
        addedAt: number; // 添加时间戳
      }>;
    }
    ```
4.  **高速目录扫描器 (High-Speed Scanner)：**
    - 当 WebUI 请求某个 Workspace 的详情时，Daemon 利用 Node.js 的 `fs.readdir` (或轻量库如 `fast-glob`) 遍历该路径下的子目录。
    - **严格排除策略：** 必须主动跳过 `node_modules`、`.git`、包含在 `.gitignore` 中的目录、以及 Git Worktree 产生的隔离目录。
    - **有效包提取：** 寻找 `package.json`，如果包配置了 `"private": true` 则直接忽略。
5.  **Profile 归属匹配 (Profile Filtering)：**
    - 提取出所有 Public 的 `package.json` 后，读取其 `name` 字段。
    - 根据当前激活的 Profile (例如当前 Profile username 为 `org-admin`)，通过包名的 Scope (`@org-admin/xxx`) 或其他匹配规则，只把属于当前身份的包列表返回给 WebUI 进行展示。

### 5.4 发包拦截与 NPM API 调度 (Publish Intercept & Scheduler)

1.  **拦截挂起：** 收到 CLI 的 Publish 意图后，Daemon 冻结发包流程，向 SvelteKit 前端发送 oRPC `state.subscribe` 投影事件。
2.  **凭证组装：** 前端通过 oRPC 回传物理确认后，Daemon 从内存提取 `@github/keytar` 解密出的 Token。
3.  **动态 TOTP：** 利用内存中的 TOTP Secret 和 `otplib`，计算当前毫秒级的 6 位 2FA 验证码。
4.  **代理执行：** 使用上述组装好的强权限头部，向 NPM API 发送底层 PUT 请求完成发包，或通过 npm `/trust` API 执行 Trusted Publishing 的新增、更新、移除，最终将成功/失败的结果通过 IPC Socket 或 Event 投影送回发起端。
5.  **身份上下文穿透 (Context Override)：**
    - 如果 CLI 传入了明确的 `--profile` 参数（如 `--profile=work`），Daemon 在生成 `Pending Event` 时，必须将该目标 Profile 强行绑定到该事件数据中。
    - 当 WebUI 收到此事件被强制唤起时，UI 必须**临时跳过/覆盖**全局的 Profile 隔离限制，在确认卡片上醒目提示用户：“当前命令指定使用 [Work] 身份进行操作”，并在实际执行时严格采用该覆盖身份的凭据，防止终端指令与 UI 选中的身份产生割裂。
