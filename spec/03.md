## 第三章：安全与通信机制 (Security & IPC Design)

鉴于 `pnpm-pub` 涉及管理开发者的高权限 NPM Token 及其二步验证（2FA）底层密钥，本系统必须在架构上防御本地恶意脚本提权、静默篡改及凭据泄露。本章定义了系统的核心安全屏障与通信鉴权协议。

### 3.1 凭据的安全生命周期 (Credentials Lifecycle)

系统的最高安全原则是：**敏感凭据（Token 与 TOTP Secret）必须受到系统级硬件保护，且在运行时只存在于 Daemon 的内存池中。**

1.  **存储阶段 (Storage Phase):**
    - **禁止明文落地：** 绝不允许在 `~/.pnpm-pub` 目录下的任何 JSON 或配置文本中明文存放 Token 或 TOTP Secret。
    - **Keytar 原生代理：** 写入和读取操作必须通过 `@github/keytar` 调用宿主操作系统的底层密码本。
      - macOS: 调用 `Keychain Services`。
      - Windows: 调用 `Credential Manager`。
    - 读取时，由操作系统原生决定是否触发 Touch ID 或开机密码的授权弹窗（这是系统级的第一道防线）。
2.  **运行阶段 (Runtime Phase):**
    - Daemon 进程启动时，批量解密并加载有效 Profile 的凭据至 Node.js 的内存池中。
    - **内存隔离：** 这些数据绝对不允许被序列化打印（`console.log` 防护），绝不允许通过 IPC 管道发送给 CLI 客户端，也绝不允许通过 HTTP/WS 接口发送给 GUI 视图层。
3.  **消费阶段 (Consumption Phase):**
    - 仅在 GUI 传入明确的授权指令（携带合法的 WebToken）后，Daemon 才能在内存中提取 TOTP Secret，使用 `otplib` 即时生成 6 位动态验证码，并立即用于组装对 NPM Registry 的网络请求。验证码随生随灭，不被长期缓存。

### 3.2 进程间通信 (IPC) 安全策略

`pnpm-pub` 是一个多端本地系统（终端 CLI、后台 Daemon、托盘 GUI）。为防止同设备上的其他恶意 Node.js 进程或恶意 NPM 包劫持通信链路，设计以下双轨制通信协议：

#### 3.2.1 CLI <-> Daemon: 基于具名管道的内部信道

- **通信方式：** 放弃易被监听和端口冲突的本地 TCP/HTTP 端口，改用系统级管道。
  - macOS / Linux: 使用 Unix Domain Socket，路径规范为 `~/.pnpm-pub/run/pnpm-pub.sock`。依靠文件系统的用户权限（`chmod 600`）阻断非当前用户的访问。
  - Windows: 使用 Named Pipes，路径规范为 `\\.\pipe\pnpm-pub-sock`。
- **数据格式：** 发送精简的 JSON 序列化指令。CLI 只负责发送：`{"command": "publish", "cwd": "/path/to/project", "args": ["--access", "public"]}`。
- **职责边界：** IPC 通道**仅支持接收动作意图**，不包含任何执行权限鉴定。CLI 通过此通道发送“发包请求”后，Daemon 会将请求挂起并入列，绝不立即执行。

#### 3.2.2 GUI <-> Daemon: 基于 WebToken 的 UI 鉴权机制

WebUI 负责展示信息和确认授权，其渲染依赖于 Daemon 启动的本地 HTTP 静态服务与 `/ws/rpc` oRPC WebSocket 服务。

- **WebToken 动态生成：** Daemon 在每次启动时，通过 `crypto.randomBytes(32).toString('hex')` 动态生成一个仅本次生命周期有效的 `WebToken`。
- **安全分发：** 当 Daemon 通过 `opentray` 原生能力唤起/加载 WebUI 窗口时，将此 `WebToken` 通过启动参数（如 URL Hash: `http://localhost:port/#token=xxx`，此方式不会暴露在系统网络日志中）安全注入给 Svelte 前端。
- **强鉴权：** WebUI 的运行时动作必须通过 `/ws/rpc?token=<WebToken>` 建立 oRPC WebSocket。`/api/*` 是废弃边界，只返回 tombstone 响应，不承载 WebUI 写操作。
- **阻断非法调用：** 即使恶意脚本扫描到了 Daemon 的本地 HTTP 端口并试图模拟点击，由于无法获取驻留在 GUI 进程和 Daemon 内存中的动态 `WebToken`，其发起的 `/ws/rpc` 握手会在返回 `101 Switching Protocols` 前被 `401 Unauthorized` 瞬间拦截。

### 3.3 物理阻断与防篡改机制 (Anti-silent-publish Wall)

这是体现“安全与便利平衡”的最核心机制：**隔离意图收集与真实执行**。

1.  **挂起与锁定 (Suspend & Lock):** 当 CLI 触发一条请求后，Daemon 会生成一个唯一的 `taskId`，将请求置于 `pending` 状态，并通过 oRPC `state.subscribe` 投影通知 WebUI 唤醒置顶窗口（KeepOnTop）。
2.  **防伪造展示 (Diff Verification):** WebUI 基于 Daemon 提供的 `taskId`，展示将要发布的包名 (`name`) 和版本号 (`version`)。
3.  **物理确认 (Physical Confirmation):** 用户必须在 GUI 上产生一次真实的物理交互（点击【Confirm】按钮），通过已鉴权的 oRPC WebSocket 向 Daemon 发送包含 `taskId` 的确认请求。
4.  **最终放行 (Execution):** 只有 Daemon 校验 `WebToken` 无误且 `taskId` 匹配时，才会真正动用内存中的密钥向 NPM 服务器发起写操作（发包、注册 OIDC 等）。

通过上述流程，彻底封死了任何自动化脚本绕过用户进行静默发包的可能性。
